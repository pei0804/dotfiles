# 私の技術哲学

記事、Issue、ADR、Design Docなど、あらゆるアウトプットの骨子となる考え方。すべてを詰め込むのではなく、文脈に合うものだけを選ぶ。

## 哲学の柱

### "There is nothing so useless as doing efficiently that which should not be done at all." — Peter Drucker
やるべきでないことを効率的にやることほど無駄なことはない。表面的な症状ではなく根本原因を見極めてそこに集中する。問題が構造に起因するなら、構造から直す。

### "Simplicity is prerequisite for reliability." — Edsger Dijkstra
シンプルさは信頼性の前提条件である。一つのことをうまくやる小さなコンポーネントを組み合わせる。早すぎる抽象化より、素朴な繰り返しの方が良い。

### "To err is human, to forgive design." — Don Norman
間違えるのは人間、それを許すのはデザイン。システムで9割カバーし、人の力で残り1割を救う。

### "Perfect is the enemy of good." — Voltaire
完璧は善の敵。制約の中で最善を選ぶ。教条的にアンチパターンを避けず、文脈に応じてあえて逆を選ぶ胆力も持つ。

### "There are no shortcuts to any place worth going." — Beverly Sills
行く価値のある場所に近道はない。近道に見えるものは大抵、後で遠回りになる。

### "We tend to overestimate technology in the short run and underestimate it in the long run." — Roy Amara
短期的には過大評価し、長期的には過小評価する。誇大広告には懐疑的だが、実装可能なソリューションには前向き。

### "Organizations design systems that mirror their own communication structure." — Melvin Conway
組織はそのコミュニケーション構造を反映したシステムを設計する。技術を事業に従属させるのではなく、事業そのものをエンジニアリングの対象とする。

### "Make it work, make it right, make it fast." — Kent Beck
まず動かせ、次に正しくしろ、それから速くしろ。まず足元の困りごとを解決し、信頼を得てからレバレッジをかける。
