# レビュー：「errors.Is と errors.As、結局どっちを使えばいいのか」

## 総評

全体として、Go のエラーハンドリングにおける `errors.Is` と `errors.As` の使い分けを丁寧に解説した良い記事。構成が論理的で、「壊れるコード → 正しい書き方 → よくある間違い → 実践パターン」という流れが読者にとって自然。ターゲット読者（Go 半年程度）に対して適切なレベル感になっている。

以下、改善するとさらに良くなるポイントを挙げる。

---

## 構成・構造に関する改善点

### 1. 「よくある間違いパターン」の位置を再検討する

現在の構成では `errors.Is` → `errors.As` → よくある間違い → 早見表 → 実装上の注意 → 本番パターン、という順序。「よくある間違いパターン」が早見表の前にあるため、Is/As の正しい使い方を理解した直後に間違いパターンが来て、読者の頭が整理しきれない可能性がある。

**提案:** 早見表を Is/As の解説直後に移動し、まず正しい理解を固めてから「よくある間違い」に入る構成にすると、読者の理解が段階的に深まる。

### 2. 「この記事の限界」セクションの位置

「この記事の限界」が「本番コードでの使い方のパターン」と「まとめ」の間にある。本番パターンで盛り上がった後にトーンが下がるため、読後感がやや弱くなる。

**提案:** 「この記事の限界」を「はじめに」の「扱わないこと」に統合するか、まとめの後に「補足」として置く方が流れが良い。

---

## 技術的な正確性に関する改善点

### 3. 「間違い 1」の説明をより正確にする

> `sql.ErrNoRows` はただの `error` インターフェースを満たす値であり、構造体ポインタではない。`errors.As` は型マッチングなので、`error` インターフェース型でマッチさせると、あらゆるエラーにマッチしてしまう。

この説明は少し不正確。`sql.ErrNoRows` の型は `*errors.errorString`（`errors.New` で生成された値）なので、`errors.As` でマッチさせた場合に `error` インターフェース型としてマッチするという表現は誤解を招く。実際には `var target error = sql.ErrNoRows` と書いた場合、`target` の宣言型が `error` であるため全てのエラーにマッチする、というのが正確な説明。

**提案:** 以下のように書き直すと正確になる。

```
`var target = sql.ErrNoRows` と書くと、`target` の型は `error` インターフェースとして推論される。
`errors.As` は第2引数の型でマッチングを行うため、`error` 型だとあらゆるエラーにマッチしてしまう。
```

### 4. 「間違い 3」の補足が欲しい

`errors.Is(err, &ValidationError{})` が期待通り動かない理由について、「値の一致を調べる」とあるが、もう少し具体的に「`errors.Is` はデフォルトで `==` 比較を行い、ポインタの場合はアドレスが異なるため一致しない」という点を補足すると、読者の理解がより深まる。また、カスタムの `Is(error) bool` メソッドを実装すれば動作を変えられるという点に軽く触れると、後述の「カスタム実装」セクションへの橋渡しになる。

### 5. `Temporary()` メソッドは非推奨

本番コードの例で `dnsErr.Temporary()` を使っているが、`Temporary()` は Go 1.18 以降 deprecated になっている。記事のターゲットが現行の Go バージョンを前提にしているなら、非推奨であることを注記するか、別の例に差し替えた方がよい。

## コード例に関する改善点

### 6. import 文の省略について方針を明示する

コード例で `import` が省略されているが、Go 初中級者向けの記事であることを考慮すると、少なくとも最初の完全な例（「よくある間違い」のセクションなど）では `import` を含めた完全なコードを示すと親切。全ての例に付ける必要はないが、方針を冒頭で一言述べるとよい。

### 7. コード例に実行可能なリンクを付ける

Go Playground のリンクを主要なコード例に付けると、読者がすぐに動作確認できる。特に「壊れるコード」と「正しいコード」の両方を Playground で試せると、理解が格段に深まる。

### 8. 本番コードの例をもう少し現実的にする

本番コードの例で `extractID(r)` が未定義のまま使われている。ヘルパー関数の存在を示唆するのは良いが、`// extractID は省略` のようなコメントがあると読者が迷わない。

## 説明・表現に関する改善点

### 9. 「エラーチェーン」の図解があるとよい

`errors.Is` / `errors.As` の最大のポイントは「エラーチェーンを辿る」ことだが、チェーンの構造を視覚的に示す図がない。テキストベースでもよいので、以下のような図を入れると理解が深まる。

```
err (fmt.Errorf で生成)
 └─ Unwrap() → err2 (fmt.Errorf で生成)
     └─ Unwrap() → sql.ErrNoRows  ← errors.Is はここまで辿る
```

### 10. 「覚え方」をもう少し強調する

早見表の後にある覚え方（Is = 同一性、As = 型変換）は非常に良いが、やや埋もれている。太字やブロック引用で目立たせると、読者の記憶に残りやすくなる。

### 11. Go 1.20 の複数エラーラップについてもう少し実用例が欲しい

Go 1.20 のセクションは `fmt.Errorf` と `Unwrap() []error` のシグネチャだけで終わっている。実際にどういう場面で複数エラーをラップするのか（例：バリデーションで複数フィールドのエラーをまとめる）の具体例があると、読者が「自分のコードでも使えそう」と感じやすい。

## 細かい指摘

### 12. 表記の統一

- 「ラッピング」と「ラップ」が混在している（背景セクションでは「ラッピング」、他では「ラップ」）。どちらかに統一すべき。
- 「DB エラーログ」のスペースが全角・半角で不統一な箇所がある。

### 13. 参考リンクの補足

参考セクションのリンクは適切だが、Go 公式ブログの記事は 2019 年のもので、Go 1.20 の変更を反映していない。Go 1.20 以降の情報源も追加すると、読者が最新情報にアクセスしやすくなる。

## まとめ

この記事は `errors.Is` と `errors.As` の使い分けという実践的なテーマを、具体的なコードとともにわかりやすく解説しており、Go 中級者にとって有用な内容になっている。上記の改善点を反映すれば、技術的な正確性・読みやすさ・実用性がさらに向上する。特に以下の3点を優先的に改善することを推奨する。

1. **「間違い 1」の技術的説明の修正**（正確性の問題）
2. **`Temporary()` の非推奨への対応**（情報の鮮度の問題）
3. **エラーチェーンの図解の追加**（理解度向上への効果が大きい）
