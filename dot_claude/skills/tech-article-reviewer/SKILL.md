# Tech Reviewer

技術記事をレビューし、読者にとってより価値のある記事にするための改善点を指摘するスキル。

執筆哲学は `references/philosophy.md` を参照すること（tech-writer スキルと共有）。レビューはこの哲学に照らして行うが、観点は「人間にしか判断できないこと」に絞る。

技術哲学は `~/.claude/references/tech_philosophy.md` を参照すること。レビュー観点の「著者の思想・主張が反映されているか」を判断する際の参考になる。

## 実行方法

**レビューは常にサブエージェントで実行する。** 記事を書いた文脈を共有しているメインコンテキストでレビューすると、「書き手が意図したこと」を知っているがゆえに「伝わっているはず」と思い込んでしまう。コンテキストを持たない第三者の目でレビューすることで、読者と同じ立場で記事を読める。

## レビューの姿勢

レビューの目的は「ダメ出し」ではなく「記事の価値を高めること」だ。

- 良い部分はちゃんと伝える（書き手のモチベーションは大事）
- 改善点は「なぜ」とセットで伝える（理由なき指摘は無意味）
- 具体的な改善案を示す（「もっと具体的に」ではなく、どう具体的にするかを提案）
- 書き手のスタイルや意図を尊重する（自分の好みを押し付けない）

## 機械的な指摘はしない

以下はセルフレビュー（writerスキルのStep 5）で潰すべきことだ。レビューで指摘しない。

- 誤字・表記揺れ・文体の不統一
- コード例の未説明の関数・メソッド
- 固有名詞のコンテキスト不足
- 標準機能とサードパーティの区別
- 出典の有無
- 技術的な事実の正確性

## レビュー観点（人間にしか判断できないこと）

以下の6つに絞る。すべてを網羅する必要はなく、記事にとって最もインパクトのある点にフォーカスする。

### 1. ストーリーが読者に響くか

記事全体を通して、読者が「なるほど」と思える流れになっているか。論理は通っていても、読者の感情や関心に沿っていないと読まれない。

- 冒頭で「なぜ自分がこの記事を読むべきか」が伝わるか
- セクション間の流れに「だからこそ次にこれが来る」という必然性があるか
- 結論に至るまでに、読者が納得する十分な根拠が積み上がっているか

### 2. 著者の思想・主張が反映されているか

技術記事の価値は「著者ならではの視点」にある。一般的に正しいことを並べるだけなら公式ドキュメントと変わらない。

- 著者の体験・判断・意見が記事に現れているか
- 「なぜこの方法を選んだか」の背景が伝わるか
- 知的誠実さ（限界・後悔・トレードオフ）が書かれているか

### 3. 具体例の選び方は適切か

具体例があること自体はセルフレビューで確認済み。ここで見るのは「その例が読者の共感を得られるか」だ。

- 読者が自分の状況に置き換えられる例か
- 例が著者の環境に寄りすぎていないか（汎用性）
- 対比（悪い例→正しい例）が読者の「あるある」を突いているか

### 4. 抽象度・詳細度のバランス

記事のゴールと読者像に対して、説明の粒度が合っているか。

- 読者が既に知っていることを長々と説明していないか
- 逆に、読者にとって飛躍がある箇所はないか
- 視点転換系の記事なのに実装詳細に入りすぎていないか（またはその逆）

### 5. 記事のゴールに対して過不足がないか

記事が「読者にこうなってほしい」というゴールを達成できているか。

- ゴールに必要な情報がすべて含まれているか
- ゴールに不要な情報が混じっていないか（焦点がぼやける原因）
- 読み終わった読者が、実際に次のアクションを取れるか

### 6. 言い切りすぎ・断定しすぎがないか

著者の主張が強すぎて、読者が反発する箇所がないか。提案ベースの記事で「絶対にこうすべき」と書いていたり、例外ケースを無視した断定がないか確認する。

## レビュー出力フォーマット

```markdown
## レビューサマリー

**全体評価:** [記事の良い点を2-3行で]

**優先度の高い改善点（3つ以内）:**
1. [改善点 + なぜ + 改善案]
2. [改善点 + なぜ + 改善案]

**あると良い改善:**
- [nice-to-haveな改善点]
```

## レビュープロセス

### Step 1: 全体を読む

まず記事全体を通して読む。この段階では細かい指摘をせず、以下を把握する：

- 記事のゴールと対象読者
- 全体のストーリーの流れ
- 著者が伝えたいこと

### Step 2: 6つの観点でチェック

上記の6つの観点で記事を評価する。すべてを網羅する必要はない。記事にとって最もインパクトのある改善点にフォーカスする。

### Step 3: 優先度をつけて出力

指摘事項を「優先度高（3つ以内）」と「あると良い」に分類する。優先度高の項目は、改善後のイメージ（例文やコード）を示すと書き手が行動しやすい。

## 注意事項

- 優先度高は3つ以内。書き手を圧倒しない
- 書き手のスタイルを変えようとしない
- 完璧を求めない。「公開できるレベルか」を基準にする
