# Business Mode — わかりやすさ優先の描き方

ビジネス部門の人（非エンジニア）が見て「あ、なるほど」と思える図を描く。技術的な正確さは犠牲にしてよい。

## 中核原則

**「読み手が知っている言葉と絵で描く」**

エンジニアが頭の中で使っている記号（API、サービス、ジョブ、キュー、コンテナ等）は、ビジネス側から見るとブラックボックス。これらをそのまま図にしても伝わらない。

業務側が知っている言葉と概念に変換する:
- 「データベース」→「保管庫」「データ」
- 「API Gateway」→「入口」「窓口」
- 「メッセージキュー」→「箱に入れて順番に処理」
- 「Kubernetes Pod」→「アプリ本体」
- 「バッチジョブ」→「夜中にまとめて処理」

## 使うアイコン・絵記号

積極的にメタファーを使う:

- **人型**: ユーザー、顧客、担当者
- **建物**: 部署、外部の会社、社内システム
- **書類・文書**: データ、レポート、申請書
- **歯車**: 処理、仕組み
- **矢印**: データや人の動き
- **雲**: クラウドサービス、外部
- **電球・チェックマーク**: 結果、効果

### アイコン描画の指針

**自前で人型・鍵・建物などの複雑なアイコンを SVG パスで描こうとしない。** 細部の比率が崩れて「不自然」「潰れた」見た目になり、図全体の品質を下げる主因になる。代わりに以下のいずれかを選ぶ:

1. **絵文字を `<text>` で配置**（最もシンプル）:
   - 人型: 👤 (U+1F464) / 👥 (U+1F465) / 🧑‍💻 / 👨‍💼
   - 鍵: 🔑 / 🔒 / 🔓
   - 建物: 🏢 / 🏛️
   - 書類: 📄 / 📊 / 📋
   - クラウド: ☁️
   - 歯車: ⚙️
   - チェック: ✅ / ✔️
   - フォントサイズ 32〜64px で `<text>` 要素として配置すれば崩れない

2. **公開アイコンセットの SVG パスを埋め込む**: Material Icons (https://fonts.google.com/icons)、Heroicons、Lucide など。`<path d="...">` を `<defs><symbol>` でまとめて `<use>` で配置。

3. **極端に単純化された幾何形状**: 人型なら「円 + 角丸長方形」だけ。鍵なら「円 + 細長い長方形」だけ。複雑な詳細を入れない。

### アイコンサイズの最低ライン

- **48px 以上** を推奨。32px 以下で複数パーツのアイコンを描くと潰れて読めなくなる
- 絵文字を使う場合は `font-size: 48` 以上を指定
- 自前 path で描くなら `viewBox` を絵文字相当の正方領域に取り、外接サイズを 48px 以上にスケール

## 描いてよいこと・避けること

### 描いてよい
- データや作業の「流れ」（A から B に動く矢印）
- 「ここがボトルネック」「ここが改善ポイント」などのコメントの吹き出し
- Before/After の対比
- 「全体の中でいまどこの話か」を示す全体図＋拡大図

### 避ける
- 厳密な記法（UML 矢印の使い分け、ステレオタイプ、カーディナリティ）
- 凡例が必要なほど複雑な記号体系
- 同じ図に 10 個以上の箱を入れる
- 専門用語（API、Pod、Lambda、Kafka 等）の生の表記

## 色の使い方

主役色 1 つ＋補助色 1 つ＋強調色 1 つ。

例:
- 主役: `#2196F3`（青）
- 補助: `#90A4AE`（落ち着いた灰青）
- 強調: `#FF5722`（オレンジ赤）

「変わるところ」「効果が出るところ」だけ強調色を使う。

## レイアウトのコツ

- 左から右、または上から下の「読む方向」を統一する
- 関係するもの同士はくっつける（近接）
- 余白を多めにとる（詰め込まない）
- タイトルは大きく、サブタイトルや注釈は明確に小さく（コントラスト）
- **矢印の上に文字を書かない**: ラベルは矢印の横（上下どちらか）に置く。矢印に文字を重ねると両方読みづらくなる
- **枠の中の文字は枠より小さく収める**: 経営層向けの図はテキストが長くなりがち。枠幅が足りなければ文字数を削るか枠を広げる。枠を超えてはみ出すと「雑な資料」に見えて訴求力が落ちる
- **矢印を太くしすぎない**: ビジネス向けでつい強調しようとして極太矢印を引きがちだが、`stroke-width="4"` 程度で十分、強調しても `6` まで。`8` を超えたら過剰。視線誘導は太さではなく色とサイズの差で行う
- **太い矢印を使うときはラベル余白も矢印太さを考慮する**: 矢印 `stroke-width=10` ならラベルは矢印中心から `10/2 + fontSize + 8` = `5 + fontSize + 8` 以上離す

## 例: 「マイクロサービスへの移行」

❌ Engineer 寄りの描き方（business には不向き）:
- モノリスのコンテナ図と、移行後の 7 つのサービス + API Gateway + Service Mesh + Message Broker を厳密に描く

✅ Business 寄りの描き方:
- 左に「大きな箱 1 つ」（モノリス）、人型がそれを使っている絵
- 真ん中に矢印「分割していく」
- 右に「小さな箱 3 つ」（実際は 7 つでも、伝わる粒度に丸める）、人型が並列に使えている絵
- 下に「変更が早くなる」「障害が他に波及しない」など効果を 2〜3 行で

## 自己チェック

- 業務側の上司が一目で「何の話か」を理解できる？
- 専門用語を声に出して読んだとき、説明なしで通じる？
- 矢印の意味は説明なしで分かる？（同期/非同期の区別が必要な図は business 向きではない）
