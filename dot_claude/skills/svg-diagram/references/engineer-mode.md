# Engineer Mode — 正確な記法で描く

エンジニアが読む前提なので、業界標準の記法を採用する。記法から意味が読み取れることが最優先。

## 中核原則

**「記法は語彙。共通の語彙を使えば説明が要らない」**

業界で確立した記法を使えば、図を見ただけで「これは同期呼び出し」「これは継承」「これは外部システム」と分かる。独自記号を使うほど、凡例の読解負荷が増える。

## 図のタイプ別の標準記法

### アーキテクチャ図 — C4 Model

- **System Context**: システム全体と外部の関係。人型 + システム箱 + 外部システム箱
- **Container**: 実行単位（Web App、DB、Worker など）の構成。箱の中に技術スタックを書く
- **Component**: コンテナ内部のコンポーネント分割
- **Code**: 通常 SVG では描かない（クラス図に任せる）

例:
```
[Person] --uses--> [Web App: React]
[Web App] --calls--> [API: Go]
[API] --reads/writes--> [DB: PostgreSQL]
```

各ボックスには `[名前: 技術スタック]` を書く。

### シーケンス図 — UML

- **ライフライン**: 縦の点線
- **活性化区間**: ライフライン上の細長い長方形
- **同期呼び出し**: 実線 + 黒塗り三角矢印
- **非同期メッセージ**: 実線 + 開き矢印
- **戻り値**: 点線 + 開き矢印
- **作成**: `<<create>>` ステレオタイプ
- **破棄**: `<<destroy>>` または X 印

時系列は必ず上から下へ。並列処理は `par` フレーム、条件分岐は `alt` フレームで囲む。

### クラス図 — UML

- **継承**: 実線 + 白抜き三角矢印（親に向く）
- **実装**: 点線 + 白抜き三角矢印
- **関連**: 実線 + カーディナリティ（1, 0..1, 0..*, 1..*）
- **集約**: 実線 + 白抜き菱形
- **コンポジション**: 実線 + 黒塗り菱形
- **依存**: 点線 + 開き矢印

可視性: `+` public, `-` private, `#` protected, `~` package。

### ER 図

カーディナリティ表記の流派が複数あるので、ひとつ選んで統一する:
- **IE (crow's foot)**: 鳥の足、輪、丸の組み合わせ
- **min-max**: `(0, N)` のように両端を書く
- **UML 表記**: `1`, `0..*`, `1..*` などをそのまま書く

主キーは下線、外部キーは `(FK)` などで明示。

### フロー図 — BPMN / Activity

- **開始**: 細い円（または塗りつぶし円）
- **終了**: 太い円（または太い枠の円）
- **タスク**: 角丸長方形
- **判断**: 菱形（出口に条件を書く）
- **並列**: 細い長方形のバー (fork / join)
- **イベント**: 円の中に絵記号（タイマー、メッセージ等）

## 矢印の意味を必ず使い分ける

| 意味 | 線種 | 矢じり |
|---|---|---|
| 同期呼び出し | 実線 | 黒塗り三角 |
| 非同期 | 実線または点線 | 開き矢印 |
| データフロー | 太い実線 | 開き矢印 |
| 継承 | 実線 | 白抜き三角 |
| 実装 | 点線 | 白抜き三角 |
| 依存 | 点線 | 開き矢印 |
| 戻り | 点線 | 開き矢印 |

混在するときは凡例を入れる。

## 色の使い方

色には意味を持たせて使う。装飾としては使わない。

例:
- **変更箇所**: `#F44336`（赤）
- **新規追加**: `#4CAF50`（緑）
- **外部システム**: `#9E9E9E`（灰）
- **ユーザー側**: `#2196F3`（青）
- **背景**: `#FAFAFA`（薄灰）

色だけで意味を伝えない（色盲対応）。形・テキスト・パターンと併用。

## 抽象度の統一

1 つの図で、抽象度（粒度）を混在させない。

❌ 悪い例:
- 同じ図に「Web Layer」（広い概念）と「UserController#login()」（メソッドレベル）を混在

✅ よい例:
- 「Web Layer」レベルで揃える、または「Controller / Service / Repository」レベルで揃える

混在したいときは、図を分けるか、ネスト表現（C4 model のように）でレベルを明示する。

## 凡例の置き方

- 右上または下部に小さく配置
- 凡例も整列・反復のルールを守る
- 凡例の枠は本体と区別できるよう薄い色や点線で囲む

## 例: 「マイクロサービスへの移行」

✅ Engineer 寄りの描き方:
- C4 Container 図で「移行前」と「移行後」を左右に並べる
- 移行前: `Monolith (Java)` 箱、その中に Controller / Service / Repository が積層
- 移行後: 7 つの Service 箱、`API Gateway (Kong)` を経由してアクセス、`Message Broker (Kafka)` で非同期連携
- 同期呼び出し（実線）と非同期メッセージ（点線）を使い分け
- 凡例: 同期 / 非同期 / 外部システム / DB の記号
