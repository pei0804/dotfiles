# SVG Diagram

SVG 形式の図解を、対象オーディエンス（ビジネス／エンジニア）に合わせて描き分けて生成する。

## 起動時にまず確認すること

呼び出されたら、以下が揃っているかを確認する。揃っていなければユーザーに質問する。

1. **モード**: `business` か `engineer` か（必須）
2. **保存先パス**: どこに `.svg` を書き出すか（必須）
3. **図のテーマ・主要要素・関係性**: ユーザーが具体的に何を描きたいか
4. **サイズ**: 指定がなければ `1280x720`（16:9）をデフォルト

モードが明示されていないときは聞き返す。推測で進めない（誤ったモードで作ると作り直しになり手戻りが大きい）。

## 2つのモード

このスキルの中核は「同じ題材でも、見る相手で描き方を変える」こと。モード別の方針は `references/business-mode.md` と `references/engineer-mode.md` を参照。判断に迷ったらそちらを読む。

### Business mode — わかりやすさ優先

ビジネス部門のステークホルダー（非エンジニア）に向けて「腹落ち」してもらうことを最優先する。

- **正確性は犠牲にしてよい**: UML の厳密な記法（菱形矢印、ロリポップ、ステレオタイプ等）は使わない。図が伝えるべきは「概念」であって「仕様」ではない
- **メタファーを積極的に使う**: 人型アイコン、建物、書類、矢印、雲などの直感的な絵記号で表現する
- **専門用語は言い換える**: 「API Gateway」→「入口」、「Kubernetes Pod」→「アプリ本体」など、業務側の言葉に近づける
- **要素数を絞る**: 3〜7 個程度。多くなる場合はグルーピングで「3つの塊」に見せる
- **色数は 3 色まで**: 主役 1 色 + 補助 1 色 + アクセント 1 色

### Engineer mode — 正確な記法

エンジニアが読むことを前提に、業界標準の記法で描く。

- **業界標準の記法を採用する**: UML、シーケンス図、ER 図、C4 model など、文脈に合うものを選ぶ
- **矢印は意味で使い分ける**: 同期呼び出し（実線＋三角矢印）、非同期（点線）、データフロー（太線）、継承（白抜き三角）など
- **凡例を入れる**: 記法が複数混在するとき、または独自記号を使うときは右上か下部に小さく凡例を置く
- **抽象度を統一する**: 1つの図の中でレイヤーや粒度を混在させない（コンポーネントとクラスを同居させない等）
- **色は意味を持たせる**: 「変更箇所は赤」「外部システムは灰」など、色に役割を割り当てて使う

どちらのモードでも、図のタイプ（アーキテクチャ、フロー、関係、比較、コンポーネント、概念）自体は共通。違いは「描き方の流儀」のみ。

## デザインの 4 原則（両モード共通）

ノンデザイナーズ・デザインブックの 4 原則を守る。詳細は `references/design-principles.md` 参照。

- **近接 (Proximity)**: 関連する要素同士は近づけ、関係のない要素は離す。グループ感を空間で表現する
- **整列 (Alignment)**: すべての要素を意図的に揃える。中央揃え、左揃え、グリッドのいずれかを選び一貫させる
- **反復 (Repetition)**: 色・形・フォントサイズなどのデザイン要素を意図的に繰り返す。同じ役割の要素は同じ見た目に
- **コントラスト (Contrast)**: 重要なものを目立たせる。差は「大きく」つける。中途半端な差は読み手を迷わせる

## 共通の技術仕様

### SVG 構造

```xml
<?xml version="1.0" encoding="UTF-8"?>
<svg xmlns="http://www.w3.org/2000/svg"
     viewBox="0 0 1280 720"
     width="1280" height="720"
     role="img"
     aria-labelledby="diagram-title diagram-desc"
     font-family="system-ui, -apple-system, 'Segoe UI', 'Hiragino Sans', sans-serif">
  <title id="diagram-title">[タイトル]</title>
  <desc id="diagram-desc">[簡潔な説明]</desc>
  <defs>
    <!-- 再利用要素（矢印マーカー等）。マーカーの path にも fill を直接書く -->
    <marker id="arr" viewBox="0 0 10 10" refX="9" refY="5" markerWidth="7" markerHeight="7" orient="auto-start-reverse">
      <path d="M0,0 L10,5 L0,10 z" fill="#475569"/>
    </marker>
  </defs>
  <!-- 1. 最初に全面の白背景を必ず敷く -->
  <rect x="0" y="0" width="1280" height="720" fill="#ffffff"/>
  <!-- 2. 以降、各要素は presentation 属性をインラインで指定する -->
  <!-- 線・矢印は必ず fill="none" を明示する -->
  <path d="M100,100 H300" fill="none" stroke="#475569" stroke-width="2" marker-end="url(#arr)"/>
</svg>
```

- `viewBox` は必ず指定する。リサイズに耐える
- `<title>` と `<desc>` は必須。スクリーンリーダーと SEO の両方に効く
- 再利用要素は `<defs>` にまとめる

### スタイルは `<style>` ではなくインライン属性で書く（必須）

`<style>` ブロック（CSS クラス）に色・線種を逃がさない。`fill` / `stroke` / `stroke-width` / `font-size` / `font-weight` などは **各要素にインライン属性として直接書く**。font-family だけはルート `<svg>` に書けば継承される。

理由は **ビューア間の互換性**。CSS の `<style>` は GitHub のサニタイズで除去されたり、スライド（Keynote・PowerPoint・Google Slides）や一部プレビューで無視されたりする。とくに線・矢印を CSS クラスの `fill: none` に頼ると、そのクラスが効かないビューアでは **開いたパスが塗りつぶされ、矢印の周りに四角い塗りが出る**（典型的な「矢印が壊れる」症状）。

- **線・矢印・コネクタには必ずインラインで `fill="none"` を書く**。これを CSS 任せにしない
- 全面の白背景 `<rect ... fill="#ffffff"/>` を最初の要素として必ず置く（透過だとダークモードや貼り付け先の背景で色が崩れる）
- マーカー（矢印頭）の `path` にも `fill` をインラインで書く

### サイズ

- デフォルト: `1280x720`（16:9、ブログ記事のヒーロー画像にちょうどよい）
- スライド向け: 同じ 16:9 で OK
- ドキュメント・README 向け: `800x600`（4:3）も検討
- 縦長フロー: `720x1280` などユーザーに確認

### 文字とコントラスト

- 最小フォントサイズ: 14px（読みやすさのため）
- テキスト色と背景色のコントラスト比は 4.5:1 以上（WCAG AA 準拠）
- 推奨フォント: `system-ui, -apple-system, "Segoe UI", "Hiragino Sans", sans-serif`

### 色の扱い

- グラデーションは原則使わない（色数を増やす割に情報量が増えない）
- 1 つの図で使う色は最大 5 色まで。意味のある色だけ
- 色だけで情報を区別しない（形・パターン・ラベルも併用 = アクセシビリティ）

### 線・矢印の引き回し（コネクタ）

矢印は「壊れて見える」事故が最も起きやすい部分。座標を置くときに次を必ず守る。

- **コネクタを他のボックスに重ねない**: 縦線・横線の通る座標が、関係ないボックスの bounding box（x 範囲かつ y 範囲）に入っていないか、線ごとに検算する。とくに縦のトランク線は、上下にあるボックスの x 範囲と重なりやすい。重なる場合はボックスの外側（左／右／上／下）に逃がす。
  - 例: ボックスが `x=980..1128` にあるなら、その手前を縦に通すトランクは `x<980` に置く（`x=1004` のように内側に置くと線がボックスを貫通し、矢印が中に刺さって見える）
- **矢印の先端はボックスの「辺」に当てる**: 入線は対象ボックスの上下左右いずれかの辺の中央付近に、辺と垂直に当てる。角や辺の端ぎりぎりに当てない。
- **多対 1 は「曲げて寄せ集める」より「まっすぐ」を優先**: 複数ソースが 1 つの箱に入るとき、各線を中央付近の数点へ曲げて寄せると線が絡む。次のどちらかにする。
  - 受け側の箱を全ソースの高さに広げ、各ソースから**まっすぐ水平**に入れる（最も素直）
  - 単一の縦バス（マニフェスト）に各ソースの短いスタブを接続し、バスから 1 本だけ矢印を出す
- **折れ線は L 字（直角 1 回）まで**を基本にし、同じ座標を通る平行線が重ならないよう間隔をあける。

## 図のタイプ別ガイド

### アーキテクチャ図
- レイヤード／マイクロサービス／イベント駆動などのパターンを選ぶ
- engineer: C4 model の Container や Component ダイアグラムの記法に寄せる
- business: 上から下、または左から右への「データの流れ」を直感で追えるレイアウトに

### フロー図
- 開始（楕円）→ 処理（角丸長方形）→ 判断（菱形）→ 終了（楕円）が標準
- business: 判断分岐を使わず直線的なフローに圧縮する選択肢もあり
- engineer: BPMN や Activity 図の記法を採用

### 関係図 (ER / クラス / シーケンス)
- engineer のみ。business には基本向かない
- ER: カーディナリティ（1, N, 0..1 等）を明示
- シーケンス: ライフライン・活性化区間・同期/非同期矢印を区別

### 比較図 (Before/After / Options)
- 縦線または横線でエリアを明確に分割
- 同じ要素は同じ位置・同じ大きさで配置（反復）
- 「変化点」だけ色を変えて目立たせる（コントラスト）

### コンポーネント図
- ボックスの中にボックスを入れて階層を表現
- 依存方向は矢印で明示

### 概念図
- 中央に「核」となる概念を置き、周辺に関連概念を放射状または同心円で配置
- business 向き

## 実行手順

1. **モードと保存先を確認する**: 不明な場合は質問してから進む
2. **要素と関係を整理する**: 何を描き、何を描かないかを宣言する
3. **レイアウトを決める**: グリッド・余白・整列方針を頭の中で固める
4. **SVG を生成する**: `<title>` `<desc>` `<defs>` を含めて書く
5. **レイアウト健全性チェック（必須・全要素）**: 出力前に以下を **すべての矢印・すべての枠・すべてのラベル** について確認する。「メインの要素だけ」では不十分（サブ要素の被り・はみ出しは後から指摘されがち）
   - テキストが矢印・図形の上に重なっていないか（中央矢印だけでなく、ボックス間の細い矢印や接続線もすべて）
   - 枠内テキストが枠の境界を超えてはみ出していないか（タイトル枠だけでなく、説明ボックス・吹き出し・凡例も含む全枠）
     - 確認方法: 枠の幅・高さに対して、テキストの推定占有領域（フォントサイズ × 文字数 + パディング）が収まるか個別に検算
   - 矢印は意味の伝達に必要な太さで、装飾的に太くなっていないか
   - 4 原則（近接・整列・反復・コントラスト）が効いているか
   - **ボックスや矢印の数が多い場合は、表形式で全要素をリスト化して 1 個ずつチェック**（例: 「矢印 5 本、枠 8 個」を表に書き出し全てに ✓ を入れる）
   - **互換性チェック**: `<style>` ブロックを使っていないか／全ての線・矢印に `fill="none"` がインラインで付いているか／全面の白背景 `<rect>` があるか
   - **コネクタ重なりチェック**: 各コネクタの通る座標が関係ないボックスの bounding box に入っていないか／矢印の先端が辺の中央付近に当たっているか（[線・矢印の引き回し](#線矢印の引き回しコネクタ) の検算を全線に対して行う）
6. **XML 妥当性検証（必須・スキップ不可）**: 保存後すぐ `xmllint --noout <ファイルパス>` を実行し、エラーが出たら必ず修正してから次に進む
   - エラー例: `Attribute X redefined`、`Opening and ending tag mismatch`、`Premature end of data`
   - パースエラーがあるとブラウザはレンダリングに失敗し、ユーザーは「画像が見れない」状態になる
   - 修正できたら再度 `xmllint --noout` を流して PASS することを確認する
7. **実ブラウザで目視検証（必須・スキップ不可）**: `xmllint` の PASS だけでは「崩れていない」とは言えない。座標の重なりや矢印の貫通は XML 妥当でも起きる。**実際のブラウザエンジン（Chromium/Blink）で PNG 化して目で見る**。`librsvg`（`rsvg-convert`）や macOS Quick Look は CSS 無視などの崩れを再現しないことがあるため、検証の主にしない。
   ```bash
   "/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" \
     --headless --disable-gpu --window-size=1280,720 \
     --screenshot=/tmp/check.png "file://$PWD/diagram.svg"
   ```
   - 生成した PNG を開き、**矢印・コネクタを 1 本ずつ**追う（必要なら端を拡大）。貫通・絡み・辺の端への突き刺さりがないか確認する
   - ユーザーが「壊れている」と言ったら、自分の手元レンダラーで再現しなくても疑うのは**自分の図の方**。まず実ブラウザで再現を試み、ダメなら見ている環境（ツール名）を聞く
8. **保存して報告する**:
   - 保存先パス
   - モード（business/engineer）
   - 図のタイプ
   - 主要要素

## アンチパターン（やってはいけないこと）

- モード未確認のまま着手する（手戻りの最大要因）
- business モードで UML の厳密な記法を使う（読み手が記号の意味で詰まる）
- engineer モードでメタファー（人型・建物アイコン）に頼る（業務フロー図に見えて専門性を欠く）
- **矢印やシンボルの上にテキストを重ねる**（読めなくなる。テキストは横にずらす、または矢印を回避する経路で引く）
- **枠内のテキストが枠からはみ出す**（テキスト幅 × フォントサイズで占有領域を見積もり、足りなければ枠を広げるかフォントを下げるか文章を短縮する）
- **強調目的で矢印を異常に太くする**（線幅 8px を超えたら過剰、業務向けは 6px 程度まで。視線誘導は太さではなく色・位置・サイズの差で行う）
- **人型・鍵・建物などの複雑なアイコンを自前で SVG パスで描く**（部品の比率が崩れて「不自然」「潰れた」見た目になる。絵文字テキストか公開アイコンセットを使う。詳細は references/business-mode.md の「アイコン描画の指針」を参照）
- **アイコンを 32px 未満で使う**（細部が潰れて意味が読み取れない。48px 以上を確保）
- 余白なしで要素を詰め込む（近接が崩れて何がグループか不明になる）
- 整列せず「だいたい揃ってる」状態で出す（雑に見える）
- 色を 6 色以上使う（意味の区別がつかなくなる）
- グラデーション・ドロップシャドウを多用する（情報量を増やさず容量だけ食う）
- `<title>` `<desc>` を省略する（アクセシビリティ違反）
- **色・線種を `<style>` の CSS クラスに逃がす**（GitHub・スライド・一部プレビューで無視され、矢印が塗りつぶされる等の崩れが出る。presentation 属性をインラインで書く）
- **線・矢印に `fill="none"` を書き忘れる**（曲がったパスがビューアによって塗りつぶされ、矢印の周りに四角い塗りが出る）
- **白背景の `<rect>` を置かない**（透過のままだと貼り付け先やダークモードで色が崩れる）
- **コネクタを関係ないボックスに重ねて通す**（線がボックスを貫通し、矢印が箱の中に刺さって見える。座標を箱の外へ逃がす）
- **多対 1 の入線を中央付近へ曲げて寄せ集める**（線が絡む。受け箱を広げてまっすぐ入れるか、縦バスに集約する）
- **`rsvg-convert` や Quick Look の表示だけで「崩れていない」と判断する**（CSS 無視などの崩れを再現しない。実ブラウザ（Chromium）で目視するまで完了としない）
