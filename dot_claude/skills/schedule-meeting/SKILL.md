# 日程調整（空き時間探索 → 予定作成）

指定メンバーの空き時間を探索し、条件に合う候補枠を提示する。ユーザーが選んだ枠で、
その場で決めたタイトルと公開レベルの予定を作成する。

## なぜこのスキルがあるか

日程調整は「全員空いてる枠を探す」「候補から選ぶ」「予定を立てる」の3手に分かれる。
このうち空き探索は `suggest_time` が各メンバーの free/busy を突き合わせて自動でやってくれる。
ただし誰をいつ呼ぶか、何という名前で公開レベルをどうするかは状況依存で、勝手に決めると
意図しない予定が相手のカレンダーに飛んでしまう。そこでこのスキルは、探索は自動化しつつ、
**枠の選択と予定の属性決定は必ずユーザーの判断を挟む** ことで安全に確定まで運ぶ。

## インプット

- 対象メンバー（メールアドレス、1人以上。必須）
  - 自分も参加する予定なら、探索対象に自分も含める（後述）
- 探索期間（いつからいつまで探すか。必須）
- 所要時間（分。指定がなければ既定 30 分）
- 営業時間・週末除外などの希望（任意。営業時間の既定は 09:30〜18:30）
- 昼休み（12:00〜13:00）は既定で候補から除外する
- タイムゾーン（任意。既定はユーザーの primary カレンダーのタイムゾーン）

引数や会話で埋まらない必須項目（特にメンバーと期間）は **推測せず聞き返す**。

## 手順

### 1. 条件を確定する

メンバー・期間・所要時間を確定する。曖昧な相対表現（「来週」「午前中」など）は
具体的な日時とタイムゾーンに落とす。不足があれば聞き返す。

- 自分も参加する予定なら、探索対象に自分の primary カレンダーを含める。
  `suggest_time` の `attendeeEmails` に `'primary'` を追加すればよい
- メールアドレスが氏名でしか渡されず特定できない場合は、`list_calendars` で
  アクセス可能なカレンダー（メールアドレス）を確認するか、ユーザーに正確な
  アドレスを聞く

### 2. 空き時間を探索する（suggest_time）

確定した条件を `suggest_time` に渡す。

```
suggest_time(
  attendeeEmails=['alice@example.com', 'bob@example.com', 'primary'],
  startTime='2026-07-06T00:00:00',
  endTime='2026-07-11T00:00:00',
  durationMinutes=60,
  preferences={
    'startHour': '09:30',
    'endHour': '18:30',
    'excludeWeekends': True,
    'pageSize': 5
  },
  timeZone='Asia/Tokyo'
)
```

- `attendeeEmails` は探索対象（free/busy を見る相手）。自分を含めるなら `'primary'`
- `preferences` は営業時間・週末除外・件数の希望。営業時間は既定 09:30〜18:30
- `suggest_time` には昼休み除外の引数がないので、**返ってきた候補のうち 12:00〜13:00
  にかかる枠は結果側で除外** してから次の提示に回す。ユーザーが明示的に昼の時間を
  希望した場合は除外しない

### 3. 候補枠を一覧提示し、ユーザーに選んでもらう

返ってきた枠を読みやすく一覧にして提示する。例:

```
空いている候補（Asia/Tokyo、60分）:
1. 7/6(月) 10:00–11:00
2. 7/6(月) 14:00–15:00
3. 7/7(火) 11:00–12:00
どれにしますか？（番号で指定。条件を変えたい場合はその旨を）
```

- **候補が0件のとき** は勝手に条件を変えず、緩める案（期間を広げる・営業時間を外す・
  所要時間を短くする・週末を含める）を提示してユーザーに選んでもらう
- 探索対象に権限のないカレンダーが含まれていた可能性があるときは、その相手の枠が
  不正確になりうる旨を添える（後述の注意事項）

### 4. タイトル・公開レベル・Meet をユーザーに確認する

枠が決まったら、予定の属性を **毎回** 確認する。推測で確定しない。

- タイトル（「カレンダー名」。例: `Alice / Bob 定例`）
- 公開レベル `visibility`: `private`（参加者のみ詳細閲覧）/ `public`（カレンダー閲覧者全員に公開）/
  `default`（カレンダーの既定）
- Google Meet を付けるか（`addGoogleMeetUrl`）

文脈から妥当な値を提案してよいが、作成前にユーザーの了承を取る。

### 5. create_event で予定を確定する

確定した枠と属性で予定を作る。

```
create_event(
  summary='Alice / Bob 定例',
  startTime='2026-07-06T14:00:00',
  endTime='2026-07-06T15:00:00',
  timeZone='Asia/Tokyo',
  attendees=[{'email': 'alice@example.com'}, {'email': 'bob@example.com'}],
  visibility='private',
  addGoogleMeetUrl=True
)
```

- `attendees` には対象メンバーを入れる。primary カレンダーに他参加者付きで作る場合、
  自分は自動で参加者に追加される
- 探索時に `'primary'` を入れていても、`create_event` の `attendees` には実在の
  メールアドレスを入れる（`'primary'` は参加者リストには使わない）

### 6. 結果を報告する

作成した予定をまとめて報告する。

- タイトル
- 日時（タイムゾーン込み）
- 参加者
- 公開レベル
- Google Meet / 予定の URL（あれば）

## 注意事項

- **候補提示なしに勝手に予定を作成しない**。必ずユーザーが枠を選ぶ
- **タイトル・公開レベルを推測で確定しない**。作成前に毎回確認する
- `suggest_time` が見られるのは、ユーザーにアクセス権がある相手の free/busy のみ。
  権限のないメンバーは予定が見えず「空いている」と誤って扱われうる。候補が想定外に
  多い・少ないときはこの可能性をユーザーに伝える
- 候補が0件でも条件を独断で緩めない。緩め方の選択肢を出してユーザーに委ねる
