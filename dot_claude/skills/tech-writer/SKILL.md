# Tech Writer

技術記事の執筆を支援するスキル。ユーザーの執筆哲学に基づき、読者にとって価値のある記事を書く。

## 執筆哲学

**最初に `references/philosophy.md` を読むこと。** 執筆の根底にある7つの信念（悪い例→正しい例、具体で語る、本番の現実、誰でも再現可能、知的誠実さ、カジュアル+厳密、文体の使い分け）が書かれている。これらを理解した上で記事を書く。

## 記事の構成

構成は記事のゴールによって変わる。テンプレートを機械的に当てはめず、ゴールに合った構成を選ぶ。

**共通で必要なもの：**
- はじめに（扱うこと/扱わないこと、前提知識、動機）
- まとめ（要点を箇条書き）
- 参考（出典）

**ゴール別の構成パターン：**

| ゴール | 適した構成 | 省略してよいもの |
|--------|-----------|----------------|
| 技術的な問題の解決策を示す | 背景→アンチパターン→正しい方法→実装手順→運用注意 | — |
| 視点や考え方の転換を促す | 背景→従来の前提→なぜ変わったか→新しい視点→具体例 | 実装手順、運用注意 |
| 技術の比較・選定を助ける | 背景→選択肢の整理→それぞれの問題→推奨→判断基準 | 実装手順の詳細 |
| 失敗・障害の知見を共有する | 何が起きたか→原因→対処→再発防止→教訓 | アンチパターン対比 |

## 執筆プロセス

### Step 1: テーマと読者の定義（最初にやる）

テーマがざっくり決まったら、**書き始める前に必ず以下を確認する。** ここが定まらないと記事の詳細度・抽象度がブレる。会話の中で既にわかっている場合はスキップしてよいが、曖昧なまま進めない。

1. **誰に届けるか** - 具体的な読者像。「データエンジニア」だけでは不十分。「dbt歴半年でカスタムマクロを書き始めた人」くらいまで絞る。読者の前提知識レベルで、説明の粒度が決まる
2. **この記事のゴール** - 読者が読み終わったあと、何ができるようになるか / 何を判断できるようになるか。ゴールが明確だと「この情報は要る/要らない」の判断軸になる
3. **何について書くか** - トピックとスコープ。スコープは広げすぎない
4. **本番経験の有無** - 実際に使った経験があるか、学習段階か。体験ベースか調査ベースかで記事の性質が変わる

### Step 2: リサーチ＆ドメイン知識の確認（重要）

**まず自分で調べる。** 技術仕様、公式ドキュメント、API リファレンス、リリースノートなど、調べればわかることは全部調べる。ユーザーに技術的な事実確認をさせるのは時間の無駄だ。

**記事で言及する技術的な挙動は、必ず公式ドキュメントやソースコードで裏を取る。** 自分の学習データからの推測で技術仕様を書かない。ドキュメントに記載がない挙動を書く場合は「ドキュメントに明記されていないが」と断りを入れるか、実際に検証した結果として書く。「それっぽいが微妙に違う」技術情報は、読者の信頼を損ない、記事全体の価値を下げる。

**ユーザーに聞くのは「ユーザーの頭の中にしかない情報」だけ。** 具体的には：

- ビジネスロジックや業務特有のルール（「うちではこう運用している」）
- 「なぜそうしているか」の背景・動機（技術的選択の裏にある事情）
- 具体的な規模感（データ量、チーム規模等）で記事の説得力に影響するもの
- 過去に試して失敗した方法（ドキュメントには載らない実体験）
- 「一般的にはAだが、あなたの環境ではBでは？」と思った場合

**質問の仕方：**
- まとめて聞く（1問ずつだと会話が冗長になる）
- 「ここは推測で書いていいですか？」と判断を委ねるのもOK
- ユーザーが「任せる」と言った部分は推測で書いてよい

### Step 3: 構成案の提示

テーマ・読者・ゴールが定まったら、構成案を作ってユーザーに提示する。下書きに入る前にここで合意を取ることで、方向性のズレを防ぐ。

**提示する内容：**
- 各セクションのタイトルと1行の説明（何を書くか）
- 記事のゴールに合った構成パターン（上の表から選択）
- 対比構造の設計（何を「悪い例」とし、何を「正しい例」とするか）
- 省略するセクションとその理由

**例：**
```
構成案:
1. はじめに — スコープと読者定義
2. 背景 — カスタム実装が生まれる理由（課題から入る）
3. 従来のコストと新しいコスト — AI時代に「教えるコスト」が加わった（視点転換の核心）
4. 実例 — 自作マクロ→標準への移行をBefore/Afterで
5. 正直な話 — 標準に寄せて困ったケースも書く
6. カスタムを残すべきケース — 判断基準の提示
7. まとめ

※ゴールが「視点の転換」なので、実装手順の詳細セクションは省略
※対比構造: カスタムだらけのプロジェクト vs 標準寄せのプロジェクト
```

### Step 4: 下書き作成

構成に沿って下書きを書く。以下に注意：

- 各セクションに具体例（コード・データ・図）を含める
- アンチパターンを先に示してから正しい方法を示す
- 本番での注意点を盛り込む
- 出典を明記する

### Step 5: セルフレビュー

下書きが完成したら、以下の観点でセルフチェックする。

**機械的に判断できることは自分で直す。ユーザーに出す前に以下をすべてクリアすること：**

- [ ] 具体例が十分にあるか
- [ ] 「悪い例→正しい例」の対比があるか
- [ ] 前提知識が明示されているか
- [ ] 本番環境での現実が反映されているか
- [ ] 出典が明記されているか
- [ ] 「強い人だけが使える」解決策になっていないか
- [ ] コード例に未説明の関数・メソッドがないか
- [ ] 用語の表記揺れがないか
- [ ] 標準機能とサードパーティの区別が正確か
- [ ] 文体が統一されているか（です/ます or だ/である）
- [ ] 技術的な挙動の記述が公式ドキュメント・ソースコードで裏取りされているか（推測で書いていないか）

**固有名詞チェック（必ず結果を書き出す）：**

記事中のすべての固有名詞（マクロ名、関数名、ツール名、プロジェクト固有の用語）を列挙し、それぞれが初出時に「何を解決するものか」の説明を伴っているか確認する。以下の形式で結果を書き出すこと。書き出さずにチェック済みとしない。

```
固有名詞チェック結果:
- `filter_partition` → ✅ 初出時に「数億レコードの差分処理のために作ったマクロ」と説明済み
- `get_warehouse` → ✅ 初出時に「開発と本番でウェアハウスサイズを切り替えるマクロ」と説明済み
- `filter_partition_for_cumulative_type` → ❌ 説明なしで登場 → 修正する
```

これらは人に指摘されるまでもないことだ。セルフレビューで100%潰す。

## 注意事項

- ユーザーが書きたい内容を尊重する。スキルの価値観を押し付けない
- ユーザーの経験や知識を引き出す質問をする
- 記事のトーンはユーザーの好みに合わせる
- 完璧を目指さない。まず出して、フィードバックを得て改善する
