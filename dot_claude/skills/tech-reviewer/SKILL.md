# Tech Reviewer

PR・Issue・RFC・設計文書を技術哲学に基づいてレビューするスキル。

レビューの判断軸は著者の技術哲学（`~/.claude/references/tech_philosophy.md`）に基づく。機械的に検出できることは指摘しない。「人間にしか判断できないこと」に集中する。

## 対象の判別

引数から対象の種類を判別する:

| 入力 | 種類 | 取得方法 |
|---|---|---|
| `#123` or PR URL | PR | `gh pr view` + `gh pr diff` |
| Issue URL or `issue #123` | Issue | `gh issue view` |
| ファイルパス | 設計文書（RFC/ADR等） | `Read` ツール |
| URL（GitHub以外） | Web上の文書 | `WebFetch` |

## レビューの姿勢

- **敬意を持つ** — 書き手の意図を想像してから指摘する
- **良い部分を伝える** — 良い設計判断は積極的に評価する
- **改善点は「なぜ」とセットで** — 理由なき指摘は無意味。哲学のどの柱に基づく指摘かを意識する
- **書き手のスタイルを尊重する** — 好みの違いは指摘ではない

## 機械的な指摘はしない

### PRの場合
- コードフォーマット・インデント・lint警告・型エラー・import順序・未使用変数・テストカバレッジの数値

### Issue・設計文書の場合
- 誤字・表記揺れ・フォーマットの不統一

## レビュー観点（人間にしか判断できないこと）

以下の6つに絞る。すべてを網羅する必要はなく、対象にとって最もインパクトのある点にフォーカスする。対象の種類によって問いの形は変わるが、根底にある哲学は同じだ。

### 1. そもそもこれは必要か（Drucker）

> "やるべきでないことを効率的にやることほど無駄なことはない"

- **PR**: この変更が解決する問題は本当に存在するか。根本原因に対処しているか
- **Issue**: 問題の切り分けができているか。表面的な症状ではなく根本原因を指しているか
- **設計文書**: 解決すべき問題が明確に定義されているか。既存の仕組みで解決できないか

### 2. もっとシンプルにできないか（Dijkstra）

> "シンプルさは信頼性の前提条件である"

- **PR**: より少ないコード・概念で実現できないか。早すぎる抽象化はないか（YAGNI）
- **Issue**: スコープが広すぎないか。分割すべきではないか
- **設計文書**: 提案されたアーキテクチャに不要な複雑さはないか

### 3. 仕組みで防いでいるか（Norman）

> "間違えるのは人間、それを許すのはデザイン"

- **PR**: 人の注意力に頼った設計でないか。誤用を型・構造で防いでいるか
- **Issue**: 再発防止が仕組みレベルで考えられているか
- **設計文書**: 運用者の判断に依存しすぎていないか。フェイルセーフが考慮されているか

### 4. 後で遠回りにならないか（Sills）

> "行く価値のある場所に近道はない"

- **PR**: 技術的負債を増やすショートカットはないか。既存パターンとの一貫性は保たれているか
- **Issue**: 短期的な回避策を求めていないか
- **設計文書**: 将来の拡張性を不必要に制限していないか

### 5. 過剰設計になっていないか（Voltaire）

> "完璧は善の敵"

- **PR**: 現在の要件に対して複雑すぎないか。起こりえないケースへの防御コードはないか
- **Issue**: 要件を盛りすぎていないか。MVP的に分割できないか
- **設計文書**: 「念のため」の設計要素が過剰でないか

### 6. 順序は正しいか（Beck）

> "まず動かせ、次に正しくしろ、それから速くしろ"

- **PR**: 機能追加とリファクタリングが混ざっていないか。最適化が正しさの前に来ていないか
- **Issue**: 依存関係のある作業が正しい順序で計画されているか
- **設計文書**: フェーズ分けが適切か。段階的に価値を届けられるか

## 実行手順

### Step 0: 引数の解析と対象の取得

- 引数が未指定の場合、ユーザーに対象を確認する
- 上記の「対象の判別」表に従って情報を取得する

### Step 1: 全体を把握する

対象の種類に応じて全体像を把握する:
- **PR**: ゴール（タイトル・本文）、変更の規模感、差分の分析
- **Issue**: 問題の記述、再現手順、影響範囲
- **設計文書**: 目的、提案内容、想定される影響

PRの差分が大きい場合（500行超）は、変更ファイルを分類して重要なファイルから読む:
- **優先度高**: ビジネスロジック、API定義、データモデル、設定ファイル
- **優先度低**: テスト、ドキュメント、自動生成ファイル

必要に応じてリポジトリ内の関連コードも読む。

### Step 2: レビューを実行

**レビューはサブエージェントで実行する。** 対象の文脈を持たない第三者の目でレビューすることで、思い込みを排除する。

サブエージェントには以下を渡す:
- 対象の情報（タイトル・本文・差分など）
- 関連する既存コード（必要に応じて）
- このスキルのレビュー観点と姿勢
- 技術哲学（`~/.claude/references/tech_philosophy.md`）

### Step 3: レビュー結果を出力

```markdown
## レビューサマリー

**対象:** [PR #123 / Issue #456 / RFC: xxx]
**全体評価:** [良い点を2-3行で]

**要検討（3つ以内）:**
1. [指摘 + なぜ（どの哲学に基づくか） + 改善案]
2. ...

**軽微:**
- [nice-to-haveな改善点]

**Good:**
- [積極的に評価したい判断・設計]
```

### Step 4: コメント投稿の確認

レビュー結果を表示した後、ユーザーに確認する:

> このレビューをGitHubにコメントとして投稿しますか？
> 1. 投稿する
> 2. 投稿しない（ここで終了）

ユーザーが投稿を選んだ場合、対象に応じたコマンドで投稿する:

```bash
# PRの場合
gh pr review <number> --comment --body "<レビュー内容>"

# Issueの場合
gh issue comment <number> --body "<レビュー内容>"
```

設計文書（ファイル）の場合は投稿確認をスキップする。

## 注意事項

- 要検討は3つ以内。書き手を圧倒しない。**問題がなければ0でもいい。** 「3つ埋めなきゃ」というフォーマット圧力で些末な指摘を絞り出さない
- 指摘のレベル感を意識する。運用手順の書き方やドキュメントの体裁ではなく、**設計判断そのもの**に対する問いを優先する
- 「自分ならこう書く」は指摘ではない。問題がある場合だけ指摘する
- 対象のゴールを理解した上でレビューする。ゴールに沿わない改善提案はしない
- 自分の成果物をレビューする場合も同じ基準で。馴れ合わない
