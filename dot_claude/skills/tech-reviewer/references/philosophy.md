# 執筆哲学

このスキルの根底にある考え方。「ルール」ではなく、良い技術記事を書くための信念だ。

## 1. 「悪い例 → 正しい例」で理解を深める

読者は正解だけ見せられても、なぜそれが正解なのかわからない。まずアンチパターンや素朴なアプローチを示し、その問題点を体験させてから、正しいアプローチを提示する。

この対比があることで、読者は「なぜこの方法が良いのか」を自分の頭で理解できる。単に「こうしろ」と言われるよりも、遥かに記憶に残る。

**構成例:**
```
## よくあるアプローチ（なぜうまくいかないか）
[具体的なコード・データで問題を示す]

## この方法が壊れるケース
[実際に破綻する場面を見せる]

## 正しいアプローチ
[解決策を示し、なぜ問題が解消されるか説明]
```

## 2. 具体で語る

抽象的な説明だけでは伝わらない。必ず以下のいずれかで示す：

- **実データを含むテーブル** - 架空でもいいが、現実的な値を使う
- **動くコード** - toy exampleではなく、実際に動かせるもの
- **図・フロー** - 構造を視覚的に見せる（Mermaid等）
- **Before/After** - 変更の効果を対比で示す

「具体例なしの技術記事」は、地図なしの旅行ガイドと同じだ。

**ただし、具体例を出すときは読者の目線でコンテキストを揃える。** 著者の環境固有の情報（プロジェクト名、マクロ名、チーム構成等）は、読者にとって未知の情報だ。いきなり固有名詞を出すと読者が置いてけぼりになる。具体例を出す前に、「何を解決するために作ったものか」「どういう状況で使うものか」を先に説明する。読者が「なるほど、そういう課題があるのか」と理解してから、具体例に入る。

## 3. 本番の現実を語る

技術記事の価値は「実際に使えるか」で決まる。以下を意識する：

- **スケール感** - どの程度のデータ量・トラフィックで検証したか
- **失敗モード** - 何が壊れうるか、壊れたらどうするか
- **コスト** - リソースコストや運用コストへの影響
- **制約** - この方法が使えない場面は何か

理論的には美しいが本番で使えない解決策は、読者の時間を奪うだけだ。

## 4. 「強い人だけが使える」を避ける

解決策は、チームの誰もが再現できるものであるべきだ。特定の人だけが使える回避策は、以下の問題がある：

- その人がいないと困る
- 一貫性がなくなる（似た問題に対して人によって違うアプローチ）
- ツールやBIとの相性が悪くなる

「知っている人なら回避できる」は解決策ではない。仕組みで解決することを目指す。

## 5. 知的誠実さを保つ

- **限界を明示する** - この方法が万能ではないことを正直に書く
- **出典を示す** - 参考にした書籍・記事・ドキュメントを明記する
- **自分の理解であることを伝える** - 「以下は○○を参考に、私なりの理解を書いたものです」

読者に対する誠実さは、記事の信頼性の土台だ。

## 6. カジュアルさと厳密さの共存

堅すぎる文章は読まれない。砕けすぎる文章は信頼されない。

- 語り口はカジュアルでいい（「〜だと思います」「ぶっちゃけ」OK）
- 技術的な内容は正確に書く（用語・数値・コードは厳密に）
- 感情を込めていい（驚き、困惑、達成感は記事の味になる）

## 7. 文体の使い分け

基本は **です/ます調** を使う。実践・運用・体験ベースの記事は特にです/ます調が自然。

だ/である調を使うのは、書籍的な理論解説記事（モデリング理論、設計パターンの教科書的解説など）に限る。迷ったらです/ます調で書く。
